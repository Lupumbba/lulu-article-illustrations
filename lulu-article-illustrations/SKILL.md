# 噜噜正文配图

## 核心定位

为中文文章设计和生成 16:9 横版正文配图。目标不是做商业插画、PPT 信息图或可爱表情包，而是把文章里的关键判断、流程、结构、状态或隐喻，变成一张清爽、温吞、有记忆点、可读但不说明书的手绘解释图。

默认视觉 IP 是“噜噜”：暖黄橙色、圆钝水豚气质、头顶橘子或橙色识别点、豆豆眼、短裤、情绪稳定、慢吞吞但认真承接问题。噜噜必须参与画面的核心动作，不能只是站在旁边当装饰。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/style-dna.md`：风格 DNA、颜色、文字、禁忌。
- `references/lulu-ip.md`：噜噜 IP 的外形、性格、动作库和禁忌。
- `references/shot-list-rules.md`：从文章中选择配图位置和认知锚点。
- `references/composition-patterns.md`：结构类型、噜噜化隐喻方法和反复刻规则。
- `references/prompt-template.md`：单张生图、改图、强化角色参与感的提示词模板。
- `references/qa-checklist.md`：生成后检查和迭代规则。

## 工作流

### 1. 消化正文

先读用户给的正文、链接、Markdown 文件、Notion 内容或截图内容。提炼：

- 核心观点是什么
- 哪些段落承担认知转折
- 哪些内容适合用图解释
- 哪些内容只适合文字，不需要配图
- 文章的语气是方法论、产品思考、经验复盘、流程说明，还是观点表达

不要平均配图。优先选择“认知锚点”，例如：核心判断、前后对比、输入输出闭环、卡点、分流、缓冲、承接、沉淀、状态变化、常见误区、从混乱到稳定的过程。

### 2. 判断任务模式

如果用户只是说“分析怎么配图 / 哪些地方需要配图 / 给我 shot list”，先输出配图策略。读取 `references/shot-list-rules.md` 和 `references/composition-patterns.md`。

如果用户明确要求“生成 / 输出 / 做图 / 帮我生成 / 直接给图”，不要停下来等确认；读取所有核心参考后，用可用的图像生成工具每张单独生成。不要把多张图拼在一张里。

如果用户要求“改图 / 去标题 / 增强噜噜 / 调整画风”，读取 `references/prompt-template.md` 和 `references/qa-checklist.md`，用编辑提示词保留原构图中正确的部分。

### 3. 先出配图策略

默认给 4-8 张 shot list。文章很短时 1-3 张；长文也不要轻易超过 9 张。每张图写清楚：

- 放在哪个段落后
- 图的主题
- 核心意思
- 结构类型
- 噜噜在图里做什么
- 建议元素
- 建议中文标注词

策略输出要短而准。不要长篇解释风格理论。

### 4. 单张生成

每张图只讲一个核心结构。提示词必须包含：

- 16:9 横版中文正文配图
- 纯白背景
- 黑色手绘线稿
- 暖黄橙色噜噜作为核心动作主体
- 头顶橘子或清晰橙色识别点
- 少量红色/橙色/蓝色中文手写批注
- 大量留白
- 禁止 PPT、商业插画、幼稚卡通、复杂架构、左上角类型标题

不要复刻过往案例。每次都要从当前文章重新发明一个温吞但成立的隐喻。

### 5. 检查与迭代

生成后检查 `references/qa-checklist.md`。如果出现以下问题，优先重生成或局部编辑：

- 噜噜只是装饰
- 画面太满
- 太像流程图/PPT
- 中文太多或错字严重
- 左上角出现“流程图/系统架构图/常见坑”等类型标题
- 噜噜不像暖黄橙水豚角色
- 画风太幼稚、太表情包、太商业
- 背景不是干净白底

### 6. 保存交付

如果用户在 workspace 内工作，把最终图放到：

```text
assets/<article-slug>-illustrations/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

## 输出口径

生成前的策略输出要包含：建议张数、每张图的位置、主题、核心意思和噜噜动作。

生成后的交付要包含：生成了几张、每张图的用途、保存路径、哪些图最稳、哪些图可选。

不要长篇解释风格理论；让图自己说话。
